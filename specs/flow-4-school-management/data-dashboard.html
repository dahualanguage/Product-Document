1 Feature Goal

ESL teachers and school admins need to know how their students are doing — not once a semester, but continuously.

Right now, schools rely on periodic standardized tests (WIDA / ELPAC) to measure English learner progress. The problem is these tests happen once or twice a year — teachers don’t get timely, actionable data in between.

This dashboard gives them that missing layer. As students use the platform (vocabulary mirroring & content mirroring), the system tracks what they understand, where they struggle, and whether they’re improving. Teachers and admins can see this data at two levels:

  • Individual student — “How is Maria doing? Is she getting better at accessing classroom language?”
  • Group / cohort — “How are my 5th grade WIDA 2 students doing in Science overall? Which vocabulary is blocking them?”

The dashboard is customizable — users pick the filters (grade, subject, proficiency level, classroom) that matter to them, and see exactly the cohort they care about.

4 Individual Student Data

Overall status rule:
≥ 60% of key indicators improving → overall improving
≥ 60% declining → overall declining
otherwise → stable

4.1 Language Access Level

How much does the student understand in the class?
We need to know if the student can access classroom instruction without support.

Metric: Classroom Language Access %

Classroom Language Access % = (Number of words understood without support ÷ Total number of words in the content) × 100

Helps teachers understand the overall level of comprehension during instruction.

4.2 Language Difficulty Profile

How often does language block understanding?
We need to know how frequently comprehension breaks down, and whether the barrier is language or concept.

Metric: Difficulty Rate (%)

Difficulty Rate (%) = (Number of difficulty events ÷ Total number of words) × 100

This answers:

  • Is the instructional material readable for this student?
  • Is the classroom language load too high?
  • Are learning barriers caused by language or by conceptual understanding?

4.3 Learning Behavior Pattern

Does the student consistently finish assigned practice?
We need to know if the student is engaged and completing work on time.

Metrics: Completion Rate & On-time Completion Rate

Completion Rate = Completed practice sessions ÷ Assigned practice sessions × 100 On-time Completion Rate = Practice sessions completed before due time ÷ Assigned practice sessions × 100
How often does the student get stuck during mirroring?
We need to know if the student struggles to process and reproduce language in real time.

Metric: Mirroring Retry Intensity

Retry Intensity = Total number of retries on mirroring items ÷ Total number of mirroring items attempted
How often does the student redo assignments to improve?
We need to know if the student self-corrects or requires re-attempts to meet expectations.

Metric: Re-attempt Rate

Re-attempt Rate = Number of assignments with ≥2 submissions ÷ Number of assignments submitted × 100

4.4 Progress Over Time

Is the student getting better, staying the same, or getting worse?
We need to track trajectory — not just a snapshot, but the direction of change over time.
Every 5 completed learning sessions → generate a progress report
  • Tag frequency trend (from Language Access %) — comprehension trajectory
  • Scaffold completion trend (from Completion Rate) — growing independence

Progress states: improving / stable / declining

4.5 Support Need Level (RTI Tier)

How much support does this student need right now?
We classify students into RTI tiers so teachers and admins can allocate resources where they matter most.

Change rate calculation:

Change Rate = (Current window average − Previous window average) ÷ Previous window average ≥ +10% change → meaningful increase ≤ −10% change → meaningful decrease between −10% and +10% → stable
TierLabelCriteria
Tier 1 Independent Language access ≥ 80% AND difficulty rate ≤ 15% AND scaffold completion ≥ 80%
Tier 2 Needs Support Language access 50%–79% OR difficulty rate 16%–35% OR scaffold completion 50%–79%
Tier 3 Intensive Support Language access < 50% OR difficulty rate > 35% OR scaffold completion < 50%
Tier Rule: If any Tier 3 condition met → Tier 3 Else if any Tier 2 condition met → Tier 2 Else → Tier 1 Evaluation: every 10 completed sessions OR every 30 days

5 Group / Cohort Data

Users can form custom cohorts using filters. Example: “I want 5th grader’s WIDA 2 students’ data from Science teacher’s class.”

5.1 Classroom Language Access Rate

How many students in this group can access instruction on their own?
We need to see the distribution of language independence across a classroom or cohort at a glance.
  • % students accessing instruction independently (Tier 1)
  • % needing scaffold (Tier 2)
  • % high risk (Tier 3)

5.2 Common Language Barriers

What specific language is blocking students the most?
We need to surface the high-frequency vocabulary and patterns that cause the most difficulty events across the cohort.
  • Top language barriers list — high-frequency vocabulary items causing difficulty events

5.3 Instruction Effectiveness Indicator

Is the instruction actually working for these students?
We need to measure whether students can complete learning tasks and whether they rely heavily on support to do so.
  • Content mirroring completion rate
  • Learning chain completion rate
  • Support usage density
  • Instruction accessibility score

5.4 Group Progress Over Time

Is this group improving, flat, or falling behind?
We need to track cohort-level trends so admins can evaluate programs and allocate resources.
  • Average access change
  • Support dependence change
  • Completion improvement
  • Growth curve visualization

5.5 Risk Distribution

How many students are at risk, and is the number growing?
We need to flag students who need intensive support and detect emerging risk clusters before they escalate.
  • Tier 1 / 2 / 3 student counts
  • High-risk cluster identification
  • New risk cases (students who moved to Tier 3)

5.6 Comparison Between Groups Future

Not in MVP. Future capability examples:

  • Proficiency Level 2 vs Level 3
  • Science vs Social Studies
  • Classroom A vs B
  • Teacher A vs B

7 Calculation Architecture

Hybrid approach: Backend pre-compute + Frontend aggregate

LayerResponsibilityWhen
Backend Compute per-student metrics: Language Access %, Difficulty Rate, Completion Rate, Retry Intensity, Re-attempt Rate On each learning session end
Backend Update Progress Trend (improving / stable / declining) Every 5 completed sessions
Backend Evaluate & update RTI Tier classification Every 10 sessions OR every 30 days
Frontend Fetch pre-computed metrics, apply filters, aggregate for group views (averages, distributions, charts) When user opens dashboard / changes filters
Why not all frontend? A school with 200+ students, each with 50+ sessions and multiple events per session, would require pulling massive raw data to the browser. Pre-computing on the backend keeps the frontend fast — it only does filter + display on already-computed metrics.

8 CTO Implementation Notes

A. System Event Definitions

EventDefinition
Difficulty eventAny user action requesting comprehension support
Word support eventTranslation / word meaning request
Mirroring retryRepeated attempt on same mirroring line without success
Assignment submissionFinal confirmed task completion

B. Learning Session Definition

A learning session = continuous interaction block ending when:

  • Assignment submitted, OR
  • 10 minutes inactivity, OR
  • User exit

C. Task Type Tagging

Each task must include:

  • scaffolded = true / false
  • task_type = vocabulary / scaffold / mirroring

D. Calculation Engine

Batch processing schedule:

  • Progress trend → every 5 sessions
  • Tier classification → every 10 sessions
  • Nightly recomputation allowed

E. Data Sufficiency Rule

  • ≥ 3 sessions required to compute trend
  • Otherwise status = insufficient data

F. Tier Stability Rule

Tier change applies only if new classification persists for 2 consecutive evaluations.
